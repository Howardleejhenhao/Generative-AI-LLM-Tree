# Technical Task Report: MCP Source Registration Foundation
**Timestamp**: 2026-04-14 07:45
**Task Objective**: Implement a registration and configuration foundation for multiple MCP (Model Context Protocol) sources, allowing the dispatcher to manage both internal and mock external tool sources.

---
## 1. Technical Implementation

### Core Design Decisions
1. **Django-based Registration**: Introduced the `MCPSource` model to provide a persistent and manageable layer for tool sources. This allows users (and future UI) to enable/disable sources, configure metadata (via JSONField), and define source types.
2. **Lazy Dispatcher Initialization**: To avoid issues with Django's startup sequence (e.g., database access during module import), I implemented a `LazyDispatcher` proxy. This ensures that the `default_dispatcher` only initializes and queries the database when it is actually used.
3. **Multi-Source Support**: Refactored `MCPDispatcher` to load all enabled sources from the `MCPSource` model. If no sources are configured, it falls back to the `InternalToolAdapter` for backward compatibility.
4. **Mock External Adapter**: Created `MockExternalMCPAdapter` to simulate an external MCP source. This validates the architecture's ability to handle multiple sources with distinct `source_id` and `source_type` attributes.
5. **Standardized Trace Metadata**: Expanded the `ToolInvocation` model and `serialize_node` service to include `source_id`. This ensures that every tool execution is traceable back to its specific source, whether internal or external.

### Architecture
- **Model Layer**: `MCPSource` stores the identity and configuration of tool sources.
- **Service Layer**: 
    - `MCPDispatcher`: Orchestrates tool listing and execution across multiple adapters.
    - `LazyDispatcher`: Provides a singleton-like interface with lazy-loading from the DB.
    - `InternalToolAdapter`: Wraps the project's internal tool registry.
    - `MockExternalMCPAdapter`: Simulates external MCP tools.
- **Integration Layer**: `node_creation.py` now queries the dispatcher to resolve tool sources for persistence in `ToolInvocation`.
