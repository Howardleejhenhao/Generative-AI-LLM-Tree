# Technical Task Report: MCP-Compatible Internal Tool Adapter Foundation
**Timestamp**: 2026-04-14 07:15
**Task Objective**: Implement an MCP-compatible internal tool adapter foundation to upgrade the existing tool system into a standardized, extensible abstraction.

---
## 1. Technical Implementation
The goal was to move away from a direct dependency on a local tool registry and move toward a Model Context Protocol (MCP) inspired architecture. This allows the system to easily integrate with external MCP servers in the future while keeping internal tools functional.

### Core Design Decisions:
- **MCP Schema Alignment**: Created a set of normalized data structures (`ToolDefinition`, `ToolInvocationRequest`, `ToolResult`) in `tree_ui/services/mcp/schema.py`. These align with the MCP `CallToolResult` specification, specifically using a list of content blocks (e.g., `text` blocks) for tool outputs.
- **Source-Agnostic Dispatching**: Implemented `MCPDispatcher` in `tree_ui/services/mcp/dispatcher.py`, which can orchestrate tools from multiple `ToolSource` implementations.
- **Backward Compatibility via Adapters**: Developed `InternalToolAdapter` in `tree_ui/services/mcp/internal_adapter.py` to wrap the existing internal registry. This ensures that current tools like `compare_branches` continue to work without modification to their core logic.
- **Standardized Trace Metadata**: Refactored `tree_ui/services/node_creation.py` to use the new dispatcher. The `ToolInvocation` persistence now captures `tool_type` (e.g., "internal") and standardized `result_payload` (JSON list of content blocks).
